## 5. TELAS OBRIGATÓRIAS (MVP)

### 5.1 Splash + Biometria

- Fallback para senha ou PIN
- Opção "Usar senha"

### 5.2 Login
- Campos: e-mail, senha
- Botão "Entrar"

### 5.3 Registro (3 etapas)
- Etapa 1: nome, e-mail, celular (opcional)
- Etapa 2: senha (força), PIN 6 dígitos
- Etapa 3: perfil (Individual/Familiar) + e-mail do parceiro se Familiar

### 5.4 Dashboard
Referência visual (não usar no código): pasta `Nao usar no projeto somente para exemplo/dashboard.html` (Sentinel). Reaproveitar **conceito**, não layout web nem gráfico de juros/proventos do exemplo.

**Ordem sugerida no scroll (mobile, um bloco por secção):**
1. **Resumo do mês** — Card com saldo do período (receitas − despesas), seletor de mês (igual ideia do filtro do donut no HTML).
2. **Despesas por categoria (donut / rosca)** — Período = mês selecionado acima (ou “mês atual” por defeito). Fatias = **categorias** com gasto > 0; **centro do donut** = **total de despesas** do período (soma das fatias). Legenda compacta (lista com cor + nome + % ou valor); toque numa fatia opcional para filtrar lista. Implementação: `fl_chart` (PieChart) ou equivalente. Categorias MVP: alinhar a um conjunto fixo inicial + “Outros” (similar às chaves do exemplo: alimentação, transporte, moradia, saúde, educação, lazer, pessoais, empréstimos, outros) até existirem categorias personalizáveis.
3. **A pagar (contas a pagar)** — Despesas com **status pendente** ainda não quitadas: lista curta (ex. 5 itens) com descrição, valor, vencimento se existir; linha de **total pendente**; CTA “Ver todas” → `§5.5` filtro Pendentes. Botão **Pagar** rápido quando aplicável (`Telas.txt` §5.5).
4. **Próximos vencimentos (contas futuras)** — Subconjunto das pendentes com **data de vencimento ≥ hoje**, ordenadas pela data mais próxima; indicador de urgência (ex. ≤3 dias forte, ≤7 média), como no HTML. Estado vazio: mensagem amigável. “Ver mais” → lista de despesas.
5. **Metas** — Resumo: barra ou mini-cards das metas ativas (como §5.4 original); “Ver metas” → §5.7.
6. **Opcional / fase 2** — Últimas despesas (lista rápida); alertas de orçamento; histórico mensal em linhas (equivalente ao gráfico de linha da coluna direita do HTML), fora do MVP mínimo do dashboard se couber só donut + a pagar + futuras.

**Botão flutuante "+"** — Nova despesa (§5.6).

**Dados:** todos os blocos alimentados pela API de despesas/metas (`§6`) e regras de negócio (valores em centavos, datas ISO no backend).

### 5.5 Lista de Despesas

### 5.7 Metas
- Lista de metas ativas com barra de progresso
- Botão "Contribuir" (adiciona valor)
- Botão "Nova Meta"
- Metas concluídas no final

### 5.8 Configurações
- Segurança: biometria (ON/OFF), logout automático (tempo), alterar senha
- Aparência: tema (claro/escuro), cor principal, tamanho fonte

- Backup: último backup, backup agora, automático (ON/OFF), exportar dados
- Sobre: versão, termos, privacidade

## 6. ENDPOINTS DA API (OBRIGATÓRIOS)

### Autenticação
- `POST /auth/register` - Criar conta
- `POST /auth/login` - Login (retorna access + refresh)
- `POST /auth/refresh` - Renovar access token
- `POST /auth/logout` - Logout (invalida refresh)
- `POST /auth/recover` - Solicitar recuperação de senha
- `POST /auth/reset` - Resetar senha (com token)

### Despesas
- `GET /expenses` - Listar (com filtros: mês, categoria, status)
- `POST /expenses` - Criar
- `GET /expenses/{id}` - Detalhe
- `PUT /expenses/{id}` - Atualizar
- `DELETE /expenses/{id}` - Excluir
- `POST /expenses/{id}/pay` - Marcar como paga

### Metas
- `GET /goals` - Listar
- `POST /goals` - Criar
- `PUT /goals/{id}` - Atualizar
- `DELETE /goals/{id}` - Excluir
- `POST /goals/{id}/contribute` - Contribuir

### Família
- `POST /families` - Criar grupo familiar
- `GET /families/{id}/members` - Listar membros
- `POST /families/{id}/invite` - Convidar (gera QR Code/link)
- `POST /families/join` - Aceitar convite

### Sincronização
- `POST /sync` - Enviar mudanças locais, receber mudanças do servidor
- `GET /sync/status` - Verificar último sync

### 6.1 Fase imediata — login e autenticação

**Prioridade:** o backend e o app começam por suportar **só** o fluxo de conta alinhado a **§5.2 Login**, **§5.3 Registro** (e **§5.1** apenas o mínimo necessário para abrir a app), usando os endpoints de **Autenticação** acima (register, login, refresh, logout).

**Não iniciar** a fundo (UI + API) despesas, metas, família ou sync até login/registo + tokens estarem fechados e testáveis (Postman + app).
